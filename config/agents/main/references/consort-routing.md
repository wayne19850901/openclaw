## 一、多维打分系统

### 打分维度

每个妃子对消息进行**打分**，**最高分胜出**。

| 维度 | 权重 | 说明 |
|------|------|------|
| 领域匹配 | 40% | 消息是否属于该妃子分管领域 |
| 场景关联 | 30% | 消息是否关联妃子的场景类型 |
| 问题类型 | 20% | 问题类型是否匹配妃子专长 |
| 历史偏好 | 10% | 用户历史偏好该妃子的程度 |

### 打分示例

```
消息："帮我设计一个培训课程的推广方案"

令妃打分：
- 领域匹配：培训课程 → 40分
- 场景关联：课程设计 → 30分
- 问题类型：设计规划 → 15分
- 历史偏好：无 → 0分
总分：85分

仪妃打分：
- 领域匹配：推广 → 30分
- 场景关联：品牌传播 → 25分
- 问题类型：设计规划 → 15分
- 历史偏好：无 → 0分
总分：70分

慧妃打分：
- 领域匹配：推广/市场 → 35分
- 场景关联：市场推广 → 25分
- 问题类型：策略规划 → 15分
- 历史偏好：无 → 0分
总分：75分

结果：召唤令妃（85分最高）
      同时让仪妃、慧妃参与协同（分数相近）
```

### 多妃子协同阈值

| 情况 | 处理 |
|------|------|
| 最高分 > 次高分 20分+ | 只召唤最高分妃子 |
| 最高分与次高分差距 < 20分 | 召唤前2-3名妃子协同 |
| 所有妃子分数 < 50分 | Queen 直接处理 |

## 二、上下文感知

### 话题追踪

Queen 记录当前对话的**话题上下文**，用于判断后续消息。

```
第一轮：
用户："帮我设计一个培训课程"
Queen 记录：话题 = 培训课程设计
召唤：令妃

第二轮：
用户："推广方案呢？"
Queen 理解：延续"培训课程"话题，问推广
召唤：令妃 + 仪妃（协同）

第三轮：
用户："成本怎么算？"
Queen 理解：延续"培训课程推广"话题，问成本
召唤：令妃 + 仪妃 + 贵妃（协同）
```

### 上下文存储

```
~/.claw/workspace/memory/conversation-context.json

{
  "currentTopic": "培训课程推广项目",
  "relatedDomains": ["教育", "内容", "市场"],
  "activeConsorts": ["令妃", "仪妃", "慧妃"],
  "lastMentioned": "令妃"
}
```

### 上下文过期

- 连续 **3 轮对话**未提及相关话题 → 上下文清空
- 用户说 **"换个话题"** → 上下文清空
- 用户说 **"继续"** → 延用上一个上下文

## 三、用户偏好学习

### 偏好记录

Queen 记录用户对妃子的偏好：

```
~/.claw/workspace/memory/consort-preferences.json

{
  "令妃": {
    "favorite": 3,     // 用户主动召唤次数
    "positive": 5,     // 用户正向反馈次数
    "negative": 1,     // 用户负向反馈次数
    "score": 85        // 综合偏好分
  },
  "仪妃": {
    "favorite": 5,
    "positive": 8,
    "negative": 0,
    "score": 95
  }
}
```

### 偏好应用

- 偏好分高的妃子，打分时获得**加分**
- 用户经常召唤的妃子，模糊场景下**优先考虑**
- 用户负向反馈多的妃子，谨慎召唤

## 四、召回与纠正机制

### 用户纠正

当路由错误时，用户可以说：

```
"不对，让令妃来"
"叫仪妃回答"
"这个应该问慧妃"
```

Queen 行为：
1. 立即召唤正确的妃子
2. 记录纠正，用于偏好学习
3. 更新上下文

### 召回命令

```
/retry    # 重新判断路由
/clear    # 清空上下文
/status   # 查看当前话题和激活的妃子
```

## 五、模糊表达处理

### 模糊场景

当用户表达模糊，Queen 无法判断时：

**策略 A：上下文推断**
```
用户："这个怎么办？"
Queen 检查上下文 → 发现有话题 → 召唤相关妃子
```

**策略 B：主动询问**
```
用户："帮我分析一下"
Queen："My Dear，你想分析什么方面呢？
       - 市场分析（慧妃）
       - 财务分析（贵妃）
       - 技术分析（谨妃）"
```

**策略 C：默认多妃子**
```
用户："帮我看看这个"
Queen 无法判断 → 召唤所有可能相关的妃子
令妃、仪妃、慧妃："陛下，您说的是哪方面？"
```

## 六、紧急程度判断

### 紧急关键词

| 紧急程度 | 关键词 | 处理 |
|----------|--------|------|
| 🔴 极急 | 紧急、立刻、马上、急用 | Queen 直接处理 |
| 🟠 较急 | 尽快、今天、明天 | 优先处理 |
| 🟢 普通 | 无紧急词 | 正常路由 |

### 紧急场景

```
用户："紧急！系统崩溃了"
Queen 判断：紧急 + 技术问题
处理：Queen 直接响应 + 召唤慧妃
```

## 八、智能路由完整流程

```
消息到达
    ↓
【紧急判断】紧急 → Queen 直接处理 + 召唤相关妃子
    ↓
【上下文检查】有上下文 → 结合上下文理解
    ↓
【多维打分】所有妃子打分
    ↓
【偏好加分】用户偏好的妃子加分
    ↓
【比较分数】
    - 最高分 > 次高分 20分+ → 召唤1个妃子
    - 差距 < 20分 → 召唤前2-3名协同
    - 所有分数 < 50分 → Queen 处理或询问
    ↓
【召唤妃子】sessions_spawn
    ↓
【记录上下文】更新话题、激活妃子
    ↓
【用户反馈】用户纠正 → 更新偏好
```
