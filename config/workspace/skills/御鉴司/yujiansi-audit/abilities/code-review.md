# 代码审查

## 概述

派遣代码审查子Agent，在问题扩散前发现并修复。

## 核心原则

```
【禁令】未经审查的代码不得合并

原因：质量问题会在后续放大
违反后果：技术债务累积，系统质量下降
```

## 何时请求审查

**强制：**
- 在子Agent驱动开发中每个任务完成后
- 完成主要功能后
- 合并到主分支前

**可选但有价值：**
- 卡住时（新视角）
- 重构前（基线检查）
- 修复复杂bug后

## 如何请求审查

### 1. 获取Git SHA

```bash
BASE_SHA=$(git rev-parse HEAD~1)  # 或 origin/main
HEAD_SHA=$(git rev-parse HEAD)
```

### 2. 派遣代码审查子Agent

提供精确的上下文：
- **实现了什么**：刚构建的内容
- **计划/需求**：应该做什么
- **BASE_SHA**：起始提交
- **HEAD_SHA**：结束提交
- **描述**：简要摘要

### 3. 处理反馈

| 级别 | 处理方式 |
|------|----------|
| 🔴 严重 | 立即修复 |
| 🟡 重要 | 继续前修复 |
| 🟢 轻微 | 记录，稍后处理 |
| ❓ 有异议 | 用技术理由反驳 |

## 审查维度

| 维度 | 检查点 |
|------|--------|
| **功能正确性** | 是否实现了需求？ |
| **代码质量** | 是否清晰、简洁、可维护？ |
| **测试覆盖** | 是否有足够的测试？ |
| **性能** | 是否有明显性能问题？ |
| **安全** | 是否有安全漏洞？ |
| **文档** | 是否有必要的注释和文档？ |

## 审查模板

```markdown
## 代码审查报告

**范围**：{BASE_SHA} → {HEAD_SHA}
**实现**：{实现了什么}

---

### 优点
- {优点1}
- {优点2}

### 问题

#### 🔴 严重
1. {问题}
   - 位置：{文件}:{行号}
   - 原因：{为什么是问题}
   - 建议：{修复建议}

#### 🟡 重要
1. {问题}
   - 位置：{文件}:{行号}
   - 建议：{修复建议}

#### 🟢 轻微
1. {问题}
   - 建议：{优化建议}

---

### 总体评估
{通过/不通过/有条件通过}

{总体评价}
```

## 与工作流集成

**子Agent驱动开发：**
- 每个任务后审查
- 在问题累积前捕获
- 修复后再继续下一个任务

**计划执行：**
- 每批（3个任务）后审查
- 获取反馈，应用，继续

**即时开发：**
- 合并前审查
- 卡住时审查
