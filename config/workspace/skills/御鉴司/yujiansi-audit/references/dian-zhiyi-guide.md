# 典质疑指南

## 质疑目的

质疑不是反对，是确保决策质量。通过系统性地质疑，避免：
- 确认偏差
- 过度自信
- 盲点遗漏
- 群体思维

## 质疑时机

### 必须质疑

- 涉及重大资源投入
- 影响系统架构
- 改变业务流程
- 安全相关决策
- 不可逆操作

### 可选质疑

- 小范围优化
- 日常运维调整
- 明确的技术选型

## 质疑技巧

### 1. 换位思考

"如果这个决策失败，最可能的原因是什么？"

### 2. 极端假设

"如果用户量增长10倍，这个方案还能工作吗？"

### 3. 历史回顾

"我们之前有没有类似的失败经历？"

### 4. 外部视角

"其他公司是怎么解决这个问题的？"

### 5. 二阶效应

"这个决策的后续影响是什么？"

## 质疑话术

| 场景 | 话术 |
|------|------|
| 风险质疑 | "最坏情况会怎样？" |
| 成本质疑 | "有没有更低成本的方案？" |
| 时机质疑 | "现在是最好的时机吗？" |
| 能力质疑 | "我们有能力执行吗？" |
| 价值质疑 | "这真的解决了核心问题吗？" |

## 质疑记录

每次质疑应记录：
- 质疑内容
- Queen的回应
- 最终决策
- 后续验证结果

用于持续改进质疑质量。
