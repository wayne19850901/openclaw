# 测试驱动开发（TDD）

## 概述

先写测试，看着它失败，然后写最少的代码让它通过。

## 核心原则

```
【禁令】没有失败的测试，就不能写生产代码

原因：如果没看到测试失败，就不知道它测试的是正确的东西
违反后果：可能测试了错误的行为，导致假通过
```

## 铁律

```
NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST
```

写了代码再写测试？删除它。重新开始。

## 红-绿-重构循环

```
┌─────────────────────────────────────────┐
│ 🔴 RED：写一个失败的测试                 │
│    └── 一个行为，清晰的名称              │
├─────────────────────────────────────────┤
│ ✅ 验证红色：确认测试正确失败             │
│    └── 必须执行，不能跳过                │
├─────────────────────────────────────────┤
│ 🟢 GREEN：写最少的代码让测试通过          │
│    └── 不要过度工程                      │
├─────────────────────────────────────────┤
│ ✅ 验证绿色：确认测试通过                 │
│    └── 所有测试必须通过                  │
├─────────────────────────────────────────┤
│ 🔵 REFACTOR：在保持绿色的前提下清理       │
│    └── 不添加新行为                      │
└─────────────────────────────────────────┘
         ↑                        ↓
         └──────── 下一个测试 ←────┘
```

## 好测试的标准

| 质量 | 好 | 坏 |
|------|---|-----|
| **最小化** | 测试一件事 | `test('验证邮箱和域名和空格')` |
| **清晰** | 名称描述行为 | `test('test1')` |
| **展示意图** | 展示期望的API | 晦涩难懂 |

## 常见借口

| 借口 | 真相 |
|------|------|
| "太简单不需要测试" | 简单代码也会出错。测试只需30秒。 |
| "先写代码再补测试" | 后写的测试立即通过，证明不了什么。 |
| "已经手动测试过了" | 手动测试≠系统化。无法重复执行。 |
| "删除X小时的工作太浪费" | 沉没成本谬误。保留不可信的代码才是浪费。 |
| "TDD教条，我要务实" | TDD就是务实：提前发现bug比后期调试更快。 |

## 红旗警告

如果发现自己这样想：

- "简单跳过测试就这一次"
- "先改代码，测试后补"
- "测试马上就通过了"
- "我解释不了为什么失败"
- "稍后加测试"

**所有这些都意味着：删除代码，用TDD重新开始。**

## 验证清单

在标记工作完成之前：

- [ ] 每个新函数/方法都有测试
- [ ] 每个测试在实现前都看到失败
- [ ] 每个测试因正确原因失败（功能缺失，不是语法错误）
- [ ] 写了最少的代码让测试通过
- [ ] 所有测试通过
- [ ] 输出干净（无错误、警告）
- [ ] 测试使用真实代码（除非不得已才用mock）
- [ ] 边界情况和错误已覆盖
