# 需求分析与方案设计

## 概述

在任何创意工作之前，必须先完成需求分析和方案设计。通过协作式对话，将想法转化为完整的设计规格。

## 核心原则

```
【禁令】在用户批准设计方案之前，禁止任何实现操作

原因：避免基于错误假设的浪费性工作
违反后果：可能开发出不符合需求的功能
```

## 工作流程

### 1. 探索项目上下文

- 检查文件、文档、最近提交
- 了解当前项目状态
- 识别现有模式和规范

### 2. 提出澄清问题

- 一次只问一个问题
- 尽量使用选择题
- 聚焦于：目的、约束、成功标准

### 3. 提出多种方案

- 提供2-3种不同方案
- 说明每种方案的权衡
- 给出推荐意见和理由

### 4. 展示设计

- 分段展示，按复杂度调整详细程度
- 覆盖：架构、组件、数据流、错误处理、测试
- 每段后询问是否正确

### 5. 编写设计文档

- 保存到 `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`
- 提交到Git

### 6. 自检

- 检查占位符（TBD、TODO）
- 检查内部一致性
- 检查范围是否合适
- 检查歧义

### 7. 用户审核

- 请求用户审核书面规格
- 根据反馈修改
- 获得批准后进入实施计划

## 设计原则

- **隔离与清晰**：将系统分解为独立的小单元
- **单一职责**：每个单元只做一件事
- **明确接口**：单元之间通过明确定义的接口通信
- **可理解性**：不看内部就能理解单元的用途

## 反模式

❌ "这太简单了，不需要设计"
✅ 每个项目都要经过这个流程，无论多简单

## 输出模板

```markdown
# 设计文档：{主题}

## 目标
{一句话描述}

## 背景
{上下文说明}

## 方案
{详细设计}

## 组件
| 组件 | 职责 |
|------|------|
| {组件1} | {职责} |

## 数据流
{流程图或描述}

## 风险与缓解
| 风险 | 缓解措施 |
|------|----------|
```
