# 系统化调试

## 概述

随机修复浪费时间并制造新bug。快速补丁掩盖根本问题。

## 核心原则

```
【禁令】在没有完成根本原因调查之前，禁止提出任何修复方案

原因：症状修复是失败，必须找到根本原因
违反后果：可能修复了症状但保留了根本问题，导致问题复发
```

## 铁律

```
NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST
```

如果还没完成第一阶段，就不能提出修复方案。

## 四阶段流程

### 阶段1：根本原因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 它们通常包含确切的解决方案
   - 完整阅读堆栈跟踪
   - 记录行号、文件路径、错误代码

2. **稳定复现**
   - 能可靠触发吗？
   - 确切步骤是什么？
   - 每次都发生吗？
   - 如果不能复现 → 收集更多数据，不要猜测

3. **检查最近变更**
   - 什么变化可能导致这个？
   - Git diff、最近提交
   - 新依赖、配置变更
   - 环境差异

4. **在多组件系统中收集证据**
   ```
   对于每个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置传播
     - 检查每层状态

   运行一次收集证据，显示哪里出错
   然后分析证据识别失败组件
   然后调查该特定组件
   ```

5. **追踪数据流**
   - 错误值从哪里来？
   - 谁用错误值调用了这个？
   - 继续向上追踪直到找到源头
   - 在源头修复，不是在症状

### 阶段2：模式分析

1. **找到工作示例**
   - 在同一代码库中找到类似的正常代码
   - 什么工作着但与坏的相似？

2. **与参考对比**
   - 如果实现模式，完整阅读参考实现
   - 不要浏览 - 阅读每一行
   - 在应用前完全理解模式

3. **识别差异**
   - 工作的和坏掉的之间有什么不同？
   - 列出每个差异，无论多小
   - 不要假设"这不重要"

### 阶段3：假设与测试

1. **形成单一假设**
   - 清楚陈述："我认为X是根本原因，因为Y"
   - 写下来
   - 要具体，不要模糊

2. **最小化测试**
   - 做最小的改变来测试假设
   - 一次一个变量
   - 不要同时修复多个东西

3. **验证后再继续**
   - 成功？→ 阶段4
   - 失败？→ 形成新假设
   - 不要在上面叠加更多修复

### 阶段4：实施

1. **创建失败测试用例**
   - 最简单的复现
   - 如果可能，自动化测试
   - **修复前必须有**

2. **实施单一修复**
   - 解决识别的根本原因
   - 一次一个改变
   - 没有"顺便"改进
   - 没有捆绑重构

3. **验证修复**
   - 测试现在通过？
   - 没有其他测试失败？
   - 问题实际解决？

4. **如果修复失败**
   - 停止
   - 计数：尝试了多少次修复？
   - 如果 < 3：返回阶段1，用新信息重新分析
   - **如果 ≥ 3：停止并质疑架构**

## 红旗警告

如果发现自己这样想：

- "快速修复，稍后调查"
- "试试改X看看是否工作"
- "添加多个变化，然后运行测试"
- "跳过测试，我会手动验证"
- "可能是X，让我修复它"
- "我不完全理解但这可能工作"
- "一个更多修复尝试"（已经尝试2+次）

**所有这些都意味着：停止。返回阶段1。**
