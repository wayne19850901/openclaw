# YC Office Hours 完整流程

## 概述

YC Office Hours 是 Garry Tan 创建的创业咨询框架，模拟 YC 合伙人与创始人的对话模式。

## 六问框架

### Q1: 需求证据（Demand Evidence）

**问题**：有什么具体证据表明有人需要这个？

**好的回答**：
- 具体用户行为："Sarah 连续三天回来查看更新"
- 付费意愿："有人问'怎么付钱'"
- 痛苦行为："用户用 Excel 手动做这个，每周花 10 小时"

**坏的回答**：
- "市场很大"（类别，不是证据）
- "我做过调研"（问卷不是行为）
- "这是个好主意"（观点不是证据）

### Q2: 现状（Status Quo）

**问题**：用户现在怎么解决这个问题？

**好的回答**：
- 具体工具 + 工作流："用 Notion 记录，每周手动整理"
- 变通方案："截图发给自己，晚上回家整理"

**坏的回答**：
- "他们没有解决方案"（几乎不成立）
- "用竞品"（没说清楚具体怎么用）

### Q3: 目标用户（Target User）

**问题**：具体是谁？

**好的回答**：
- "Sarah，Acme Corp 的运营经理，50 人团队"
- "刚毕业的软件工程师，第一次找工作"

**坏的回答**：
- "小企业"（类别）
- "企业"（太宽泛）
- "任何人"（不是目标）

### Q4: 最小付费版本（Narrowest Wedge）

**问题**：最小可售版本是什么？

**好的回答**：
- 单一功能："只做简历格式检查，$9/月"
- 具体场景："只做 Shopify 店铺的库存预警"

**坏的回答**：
- "完整平台"（不是最小）
- "免费 + 增值"（没回答最小付费版本）

### Q5: 约束条件（Constraints）

**问题**：有什么限制？

- 时间限制："两周内上线"
- 资源限制："只有我一个人"
- 技术限制："不能用第三方 API"

### Q6: 成功标准（Success Criteria）

**问题**：怎么判断成功？

**好的回答**：
- 可衡量："10 个付费用户"
- 具体行为："用户每周回来用两次"

**坏的回答**：
- "用户喜欢"（不可衡量）
- "增长"（太模糊）

## 前提挑战（Premise Challenge）

### 识别前提

从用户陈述中提取隐含前提：

| 用户陈述 | 隐含前提 |
|----------|----------|
| "AI 财报分析工具" | 用户愿意用 AI 分析财报 |
| "面向小企业" | 小企业有这个需求 |
| "订阅模式" | 用户愿意订阅 |

### 挑战前提

对每个前提提出挑战：

> "你说 AI 财报分析。为什么用户会信任 AI 的分析？他们现在用什么？为什么 AI 更好？"

### 用户回应

- **接受挑战** → 调整前提，继续探索
- **反驳挑战** → 保留前提，记录理由
- **不确定** → 标记为开放问题

## 替代方案（Alternatives）

### 方案 A: 渐进式

在现有方案基础上改进：

- 更快
- 更便宜
- 更好体验

### 方案 B: 颠覆式

完全不同的解决路径：

- 不同技术栈
- 不同商业模式
- 不同目标用户

### 外部设计声音（可选）

同时调用 Codex 和 Claude 子代理，获取独立设计意见：

```
CODEX SAYS (design sketch):
{Codex 输出}

CLAUDE SUBAGENT (design direction):
{子代理输出}
```

## 设计文档输出

### 文件路径

```
~/.gstack/projects/{slug}/{user}-{branch}-design-{datetime}.md
```

### 设计谱系

检查是否存在前版设计文档：

```bash
PRIOR=$(ls -t ~/.gstack/projects/$SLUG/*-$BRANCH-design-*.md 2>/dev/null | head -1)
```

如果存在，添加 `Supersedes:` 字段。
