# CEO 计划评审四模式详解

## 概述

CEO 计划评审是创始人级别的战略评审，根据计划野心程度选择四种模式之一。

## 四种模式

### EXPANSION（范围扩展）

**触发条件**：
- 野心功能
- 新产品方向
- 平台化机会

**核心问题**："这是否足够大？"

**评审重点**：
- 10x 检查：这个版本是否是 10 星产品的 10%？
- 平台潜力：是否创造其他功能可利用的能力？
- Phase 2/3 规划：后续阶段是什么？

**额外输出**：
- CEO 计划文档（愿景扩展）
- 范围扩展提案列表
- 用户选择仪式（opt-in）

### SELECTIVE EXPANSION（选择性扩展）

**触发条件**：
- 有诱惑力的扩展
- 用户可能想要更多
- 不确定是否值得

**核心问题**："这是否正确 + 还有什么值得？"

**评审重点**：
- 樱桃挑选：哪些扩展值得接受？
- 风险检查：接受的扩展是否改变风险画像？
- 品味校准：是否符合设计系统？

**额外输出**：
- CEO 计划文档（选择性扩展）
- 樱桃挑选仪式
- 风险重新评估

### HOLD SCOPE（保持范围）

**触发条件**：
- 范围合理
- 时间敏感
- MVP 验证

**核心问题**："这是否过于复杂？"

**评审重点**：
- 复杂性检查：能否简化？
- 关键决策：哪些决策必须现在做？
- 回滚计划：如果失败怎么办？

**输出**：
- 评审报告（无 CEO 计划）
- 风险评估
- 部署计划

### SCOPE REDUCTION（范围削减）

**触发条件**：
- 范围过大
- 时间紧
- 资源有限

**核心问题**："这是否是最低限度？"

**评审重点**：
- MVP 定义：最小可行版本是什么？
- 削减清单：哪些功能可以砍？
- 增量交付：能否分阶段交付？

**输出**：
- 削减后的计划
- 分阶段交付计划
- 延期功能列表

## 11 个评审章节

| 章节 | 关注点 | EXPANSION 特有 |
|------|--------|----------------|
| 1. 架构 | 模块边界、依赖方向 | 平台潜力评估 |
| 2. 错误处理 | 异常映射、救援路径 | 混沌场景 |
| 3. 安全 | 认证、授权、注入 | — |
| 4. 数据/UX | 边界值、并发 | — |
| 5. 代码质量 | 命名、重复 | — |
| 6. 测试 | 金字塔、覆盖率 | 混沌测试 |
| 7. 性能 | N+1、索引、缓存 | — |
| 8. 可观测性 | 日志、指标、追踪 | 运维愉悦度 |
| 9. 部署 | 迁移、回滚 | 部署基础设施 |
| 10. 长期轨迹 | 技术债、可逆性 | Phase 2/3 规划 |
| 11. 设计/UX | 信息架构、交互状态 | UI 不可避免性 |

## 范围扩展提案格式

```markdown
## Scope Expansion Proposals

### Proposal 1: {title}
- **What**: {描述}
- **Why**: {价值}
- **Effort**: {估算}
- **Risk**: {风险评估}
- **Recommendation**: Accept / Defer / Skip

### Proposal 2: {title}
...
```

## 樱桃挑选仪式（SELECTIVE EXPANSION）

```
用户选择：
  A) Accept all proposals
  B) Cherry-pick: [列出每个提案]
  C) Defer all to TODOS.md
  D) Skip — keep original scope
```

## 外部意见（Outside Voice）

可选调用 Codex 或 Claude 子代理获取独立评审：

```bash
codex exec "<plan review prompt>" -C "$_REPO_ROOT" -s read-only -c 'model_reasoning_effort="high"'
```

输出格式：
```
CODEX SAYS (plan review — outside voice):
════════════════════════════════════════════════════════════
{完整 Codex 输出}
════════════════════════════════════════════════════════════
```

## 跨模型张力处理

当外部意见与评审发现不一致时：

```
CROSS-MODEL TENSION:
  [主题]: 评审说 X。外部意见说 Y。
  [缺失的上下文可能改变答案]
```

提交用户选择：
```
A) Accept outside voice recommendation
B) Keep current approach
C) Investigate further
D) Add to TODOS.md
```
