# 计划设计评审

## 概述

设计师之眼计划评审：交互式评审，像 CEO 和 Eng 评审。评分每个设计维度 0-10，解释如何达到 10 分，然后修复计划。在计划模式下工作。

**来源**: gstack/plan-design-review

## 使用方式

```bash
/plan-design-review <plan-file>
```

## 工作流程

```
Phase 1: 读取计划
    ├── 读取计划文件
    ├── 提取 UI/UX 范围
    └── 识别设计决策
    ↓
Phase 2: 评审设计维度
    ├── 信息架构
    │   └── 用户看到的第一、第二、第三
    ├── 交互状态覆盖
    │   └── LOADING / EMPTY / ERROR / SUCCESS / PARTIAL
    ├── 响应式意图
    │   └── 移动端是否提及
    ├── 无障碍基础
    │   └── 键盘导航、屏幕阅读器、对比度
    └── AI slop 风险
        └── 是否描述通用 UI 模式
    ↓
Phase 3: 评分与解释
    ├── 每个维度 0-10 分
    ├── 解释当前分数原因
    └── 解释如何达到 10 分
    ↓
Phase 4: 修复计划
    ├── 添加缺失的设计决策
    ├── 补充交互状态
    └── 明确响应式意图
    ↓
Phase 5: 输出报告
    ├── 评分表
    ├── 改进建议
    └── 修复后的计划片段
```

## 执行指令

**加载原始 gstack SKILL.md**:

```bash
# 读取完整技能定义
source_content=$(cat ~/.claude/skills/gstack/plan-design-review/SKILL.md)
```

将 `source_content` 作为系统提示词的一部分，执行完整的计划设计评审流程。

## 设计维度评分

| 维度 | 0-3 分 | 4-6 分 | 7-9 分 | 10 分 |
|------|--------|--------|--------|-------|
| **信息架构** | 无结构 | 基本层级 | 清晰流程 | 用户旅程连贯 |
| **交互状态** | 只有成功 | 有加载/错误 | 覆盖大部分 | 全状态覆盖 |
| **响应式** | 未提及 | 有提及 | 有具体方案 | 移动优先 |
| **无障碍** | 未提及 | 有提及 | 基础覆盖 | WCAG AA |
| **AI slop** | 通用描述 | 部分具体 | 大部分具体 | 完全具体 |

## 输出格式

```markdown
## 计划设计评审报告

**计划**: {plan-file}
**时间**: {timestamp}

### 维度评分

| 维度 | 分数 | 如何达到 10 分 |
|------|------|----------------|
| 信息架构 | {n}/10 | {建议} |
| 交互状态 | {n}/10 | {建议} |
| 响应式 | {n}/10 | {建议} |
| 无障碍 | {n}/10 | {建议} |
| AI slop | {n}/10 | {建议} |

### 改进建议

1. {建议1}
2. {建议2}

### 修复片段

{需要添加到计划的设计决策}
```

## 禁令

- **禁止修改代码** — 计划评审，只修改计划文件
- **禁止跳过维度** — 必须评审所有 5 个维度
- **必须解释 10 分** — 每个维度必须解释如何达到 10 分

## 与 design-review-lite 的区别

| 特性 | plan-design-review | design-review-lite |
|------|-------------------|-------------------|
| 输入 | 计划文件 | 实时 URL |
| 阶段 | 实现前 | 实现后 |
| 输出 | 设计改进建议 | 修复报告 |
| 修复 | 修改计划 | 直接修改代码 |
