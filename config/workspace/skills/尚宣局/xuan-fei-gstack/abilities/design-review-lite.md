# 设计评审（实时）

## 概述

设计师之眼 QA：发现视觉不一致、间距问题、层级问题、AI slop 模式、慢交互 — 然后修复。迭代修复源代码问题，原子提交每次修复，用前后截图验证。

**来源**: gstack/design-review

## 使用方式

```bash
/design-review <url>
```

## 工作流程

```
Phase 1: 启动浏览器
    ├── 启动 headless Chromium
    └── 导航到目标 URL
    ↓
Phase 2: 截取快照
    ├── 全页截图
    ├── 视口截图
    └── 元素截图（可选）
    ↓
Phase 3: 分析问题
    ├── 视觉不一致
    │   └── 颜色、字体、间距不统一
    ├── 层级问题
    │   └── 信息架构混乱、重点不突出
    ├── AI slop 模式
    │   └── 过度通用、缺乏品味
    └── 慢交互
        └── 响应时间 >100ms
    ↓
Phase 4: 迭代修复
    ├── 定位源代码
    ├── 应用修复
    ├── 原子提交
    └── 验证修复（前后截图）
    ↓
Phase 5: 输出报告
    ├── 修复列表
    ├── 前后对比图
    └── 提交哈希
```

## 执行指令

**加载原始 gstack SKILL.md**:

```bash
# 读取完整技能定义
source_content=$(cat ~/.claude/skills/gstack/design-review/SKILL.md)
```

将 `source_content` 作为系统提示词的一部分，执行完整的设计评审流程。

## 问题检测清单

| 类别 | 检查项 |
|------|--------|
| **视觉一致性** | 颜色、字体、间距、圆角、阴影 |
| **层级** | 标题大小、对比度、视觉权重 |
| **间距** | 元素间距、边距、留白 |
| **AI slop** | 过度通用描述、缺乏具体设计意图 |
| **交互** | 响应时间、动画流畅度 |

## 输出格式

```markdown
## 设计评审报告

**URL**: {url}
**时间**: {timestamp}

### 发现问题

| # | 问题 | 位置 | 修复 |
|---|------|------|------|
| 1 | {描述} | {文件:行号} | {修复方案} |

### 修复验证

**Before**:
![before]({before-screenshot})

**After**:
![after]({after-screenshot})

### 提交

- {commit-hash}: {commit-message}
```

## 禁令

- **禁止修改代码** — 评审技能，只报告问题
- **禁止跳过验证** — 每次修复必须验证
- **必须原子提交** — 每个修复单独提交

## 与 plan-design-review 的区别

| 特性 | design-review-lite | plan-design-review |
|------|-------------------|-------------------|
| 输入 | 实时 URL | 计划文件 |
| 阶段 | 实现后 | 实现前 |
| 输出 | 修复报告 | 设计改进建议 |
| 修复 | 直接修改代码 | 修改计划 |
